# 01. システム哲学と運用 (System Philosophy and Operations)

> **"複雑さは実行の敵である。機能の純粋さこそがその解毒剤である。"**

Tensor-Link Utility (TLU) は、モノリシックで状態を持ち、極度に絡み合ったアプリケーションを構築するという現代のトレンドを意図的に拒絶するように設計されています。代わりに、**Unixの哲学**に根ざした厳格で古典的なアプローチを採用しています。

このドキュメントでは、システム（Ver 8.0.0）を支配するコアなアーキテクチャの原則と運用プロトコルについて概説します。

---

## 1. Unix パイプライン・アーキテクチャ

TLU は、組織のダイナミクスを文字通り「ストリーム（流れ）」として扱い、情報の連続的なフローとしてデータを処理します。

* **ステートレス・フィルター:** 各数学フィルター（例：熱力学、運動学）は独立した実行可能ファイルです。これらは `Standard Input (stdin)` から標準の COO（座標）テンソルストリームを読み込み、物理学の単一のパラダイムを適用し、その結果を新しい列として追加して `Standard Output (stdout)` にフラッシュします。
* **隠し状態の排除:** フィルターは、実行間で内部データベースやメモリを保持しません。入力ストリームが同一であれば、出力も同一であることが数学的に保証されます。
* **パイプ接続 (`|`):** 複雑なクラスを記述することによってではなく、シンプルで純粋な関数を連結することによって、複雑な分析が達成されます。

## 2. 関心の分離：I/O vs. 純粋数学

テスト容易性（テスト駆動開発による）を最大化し、開発者の認知負荷を軽減するために、TLU はデータ処理と数学的計算の間に厳格な物理的分離を強制します。

* **ラッパー層 (`filter_*.py`):** これらのスクリプトは意図的に「汚く（dirty）」作られています。CLI 引数のパース、`sys.stdin` からの CSV の読み込み、ドメイン辞書の解決、および `sys.stdout` への書き込みを処理します。**この層には物理学、幾何学、あるいはビジネスロジックを含めることは一切許可されていません。**
* **コア数学層 (`core_*.py`):** これらは純粋な数学関数です。CSV、ドメインの語彙、ファイルシステムについては何も知りません。純粋な NumPy 配列（`np.ndarray`）とスカラーを受け取り、NumPy 配列を返します。これにより、ユニットテストが極めて容易になり、数学的厳密性が保証されます。

## 3. ローカル依存性ゼロ

「私のマシンでは動く」という言い訳は、高精度な分析環境においては容認できません。

* **ホストの無菌化:** ユーザーのホストOSに必要なのは `Docker` と `bash` のみです。Python、NumPy、SciPy、NetworkX、および Matplotlib は、`tlu-engine` コンテナ内に厳密に閉じ込められています。
* **透過的なラッパー (`_tlu_env.sh`):** TLU は、ローカルコマンドをコンテナ化された実行（例：`docker compose exec -T tlu-engine python3`）へと透過的に翻訳するシェル・ラッパーを提供します。データは標準ストリームを介してホストのファイルシステムと隔離されたコンテナ間をシームレスに流れ、コンテナをまるでネイティブ・バイナリのように感じさせます。

## 4. 宣言的実験制御（SSOT）

複雑な企業のシミュレーションにおいて、洞察の「来歴（プロベナンス）」は洞察そのものと同じくらい重要です。再現性を保証するため、TLU は Single Source of Truth (SSOT: 信頼できる唯一の情報源) に依存しています。

* **`_sys_params.csv` パラダイム:** 実験条件（例：入力データソース、ターゲットシミュレーションノード、異常閾値）を変更するために、シェルスクリプトやPythonのソースコードをユーザーが変更することは厳しく禁じられています。すべての境界条件と制約は、`workspace/config/_sys_params.csv` で宣言的に定義されます。
* **コードとしてのパラメータ:** この設定ファイルは、実験の決定的な「設計図（ブループリント）」として機能します。パラメータを実行ロジックから隔離することで、システムは意図しない設定のドリフトを防ぎます。

## 5. イミュータブル・アーカイブの再現性

外部の Git リポジトリに依存することなく完全な監査可能性を強制するために、TLU はローカルのスナップショット・アーキテクチャを採用しています。

* **ワークスペースのスナップショット:** 重要なパイプラインの実行後、以下のコマンドを実行するだけで、（入力ストリーム、使用された正確な `_sys_params.csv`、およびすべての出力データを含む）`workspace/` ディレクトリ全体を `archives/run_YYYYMMDD_HHMMSS/` ディレクトリにスナップショットとして保存できます：
  ```bash
  bash bin/archive_experimental_run.sh
  ```
  *(注：出力プロットは決定論的に再生成できるため、ディスクスペース節約のためにアーカイブからは除外されます)*。
* **タイムトラベル実行:** パイプラインはターゲット環境に対して相対的に設定を動的参照するため、ユーザーは過去の計算を再現したり、ダッシュボードを再生成したりすることが簡単にできます。オーケストレーション・スクリプトを過去のアーカイブに向けることで、パイプラインはその歴史的なスナップショットの正確な瞬間と全く同じように振る舞います：
  ```bash
  # 過去の実験から3D可視化ダッシュボードを再生成する
  bash bin/batch_visualize_graphs.sh dark --target_env archives/run_20260425_094806/workspace
  ```
